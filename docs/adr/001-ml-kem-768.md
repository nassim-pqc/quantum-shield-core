# ADR-001: Choix de ML-KEM-768 (Kyber768)

## Statut
Accepté

## Contexte
Le projet nécessite un mécanisme d'encapsulation de clé (KEM) résistant à l'informatique quantique pour l'échange de clés symétriques.

## Décision
Utiliser ML-KEM-768 (standardisé par le NIST, issu de Kyber768) comme KEM.

## Justification
- ML-KEM-768 est le standard NIST FIPS 203 pour la cryptographie post-quantique
- Niveau de sécurité NIST 3 (AES-192 équivalent)
- Ratio sécurité/performance optimal pour un microservice
- Supporté par `liboqs` (Open Quantum Safe) en Python et Rust
- 1184 octets de clé publique, 2400 octets de clé secrète

## Conséquences
- Dépendance à la bibliothèque C `liboqs` via `liboqs-python`
- Impossible d'utiliser le module Rust pour la génération de clés sans lier `liboqs`
- La clé partagée dérivée via KEM est hashée avec SHA-256 avant utilisation AES-GCM

## Alternatives rejetées
- ML-KEM-512 : niveau sécurité NIST 1 jugé insuffisant
- ML-KEM-1024 : overhead réseau trop important (1568 octets de clé publique)
- ECDH classique : pas de résistance post-quantique
